Lesson 6

How Memory Works

Memory is what separates a conversation from a relationship.

📖 Story

You've been talking to your rappter for days. You've told it your name, your projects, your preferences. But here's the question that matters: does it remember yesterday?

Most AI tools don't. Close the tab, the conversation evaporates. Open it again tomorrow and you're introducing yourself all over again. It's like meeting someone with permanent amnesia — every encounter is the first.

Your rappter is different. It has a memory system — a real one, writing to real files on your real hard drive. And that changes everything.

The Memory Architecture

Memory in the brainstem is handled by local_storage.py — a storage manager that reads and writes JSON files to the .brainstem_data/ directory sitting right next to the brainstem code.

.brainstem_data/ ├── shared_memories/ │ └── memory.json ← What the rappter remembers about YOU │ ├── memory/ │ └── {user-guid}/ │ └── user_memory.json ← Per-user memories (multi-user mode) │ ├── journal.json ← The heartbeat's inner monologue └── vitals.json ← Current heartbeat state

The key file is shared_memories/memory.json. This is where facts about you — your name, your preferences, your projects — get stored as plain JSON. No vector database. No embedding model. Just a file on your disk that you can open and read.

How Memories Get Created

The memory agent listens during conversations. When you say something worth remembering — "my favorite language is Python" or "I'm working on Project Atlas" — the agent writes it to memory.json using the AzureFileStorageManager shim in local_storage.py.

Next time you talk to the rappter — even in a brand new session — it reads memory.json and knows your favorite color is blue. The memory persists because it's a file, not a variable. Restart the server, reboot your machine — the file is still there.

The Storage Manager

The AzureFileStorageManager class in local_storage.py is a local-first shim. It mirrors the same API that agents use in the cloud (Azure File Storage), but writes everything to .brainstem_data/ on your disk instead. This is the import shim trick — agents don't even know they're running locally. 

class AzureFileStorageManager:
    def read_json(self, file_path=None):
        # Reads from .brainstem_data/shared_memories/memory.json
        path = file_path or self._file_path()
        with open(path, "r") as f:
            return json.load(f)

    def write_json(self, data, file_path=None):
        # Writes to .brainstem_data/shared_memories/memory.json
        path = file_path or self._file_path()
        with open(path, "w") as f:
            json.dump(data, f, indent=2)

It also supports per-user memory paths at .brainstem_data/memory/{guid}/user_memory.json for when multiple users share a single brainstem. But locally, most rappters use the shared path.

  -d '{"user_input": "Remember that my favorite color is blue"}'

curl -s localhost:7071/chat -X POST \
  -H "Content-Type: application/json" \
  -d '{"user_input": "What is my favorite color?"}'

cat ~/.brainstem/src/rapp_brainstem/.brainstem_data/shared_memories/memory.json

There it is — your fact, sitting in a plain JSON file on your disk. No cloud. No database. Just a file you can read, edit, or delete. Your rappter's memory belongs to you.
